# Jira 라이팅 규칙 — 설계 문서

- 날짜: 2026-06-19 (KST)
- 상태: 승인됨 (PROPOSED → APPROVED)
- 관련 파일: `rules/jira-writing.md`(신규), `core/CLAUDE.md`, `commands/dct-complete.md`, `skills/dct-jira-workflow/SKILL.md`

## 배경 / 문제의식

AI 라이팅으로 Jira에 작업을 정리하는 현재 방식에 대한 문제 제기에서 출발한다.

- 공유 문서가 한 줄도 이해 안 갈 만큼 보기 힘들다.
- 현재 방식 = 지라 티켓에 **다시 보지 않을, AI도 다시 읽지 않을 세션 히스토리를 그냥 정리**하는 방식. 유효한가?
- 협업 대상이 이해 못 하는 문서를 빠르게 적고, 결국 두세 번 다시 말하게 하는 게 정말 생산성인가?
- 그렇다고 예전처럼 직접 다 쓰는 것도 답은 아니다.

→ **Jira에 정리할 때 규칙을 정해 가독성 좋게 쓰도록 개선한다.**

## 핵심 진단

이미 `dct-complete` / `dct-jira-workflow`에 정교한 포맷 규칙(마커 고정, bullet 3~5개 등)이 있다.
그런데도 "보기 힘들다"는 건 **포맷의 문제가 아니라 "무엇을 쓰느냐(내용 선택)"의 문제**다.

페인포인트 (사용자 확인):
1. 완료 댓글이 세션 로그처럼 장황 — 결론·영향이 안 보임.
2. description 플랜이 AI스러움 — 기계적이라 협업자가 안 읽음.

독자가 원하는 것 (규칙의 1순위 기준):
- **왜 이렇게 했나(판단 근거)**
- **나중에 검색해 다시 찾을 레퍼런스**

→ 현재 포맷은 *"무엇을 바꿨나(변경 요약 + 영향 범위)"* 중심이라 위 두 가지가 약하다. 여기가 핵심 갭.

## 접근 방식 (확정)

공용 **`rules/jira-writing.md`** 파일에 라이팅 원칙·포맷을 단일 소스로 정의하고,
기존 커맨드/스킬은 이를 **참조**한다. (단일 소스 + 일관성)

근거를 녹이는 방식은 **A안 채택**: 마커를 새로 추가하지 않고 bullet 문장 자체를 `<무엇> — <왜/효과>`로 강제.
판단이 갈렸을 때만 `💡 결정:` 한 줄을 선택적으로 추가. (마커 신설 = 또 장황해지는 역설 회피)

## 규칙 파일 뼈대 (`rules/jira-writing.md`)

### 원칙 5
1. **결론 먼저** — 첫 줄은 결과("무엇이 어떻게 됐다"). 과정·배경 narration 금지.
2. **왜 > 무엇** — 변경 bullet은 `<무엇> — <왜/효과>` 한 문장. 코드 보면 아는 "무엇"만 적지 않는다.
3. **검색되게** — 첫 줄·bullet에 고유명사(모듈/기능/에러명) 포함. 카드 제목 복붙 금지.
4. **스캔 가능** — 마커·순서 고정, 산문체 금지. 5초 안에 구조 파악.
5. **세션 로그 ≠ 문서** — AI가 한 모든 일을 옮기지 않는다. PR에서 확인 가능한 것(변경 파일 목록, diff stat 등)은 댓글에 중복하지 않는다. 독자가 행동/판단에 쓸 것만 남긴다.

### 안티패턴 (하지 말 것)
- 세션 히스토리 나열 / 도구 호출·내부 사고 narration
- 매번 다른 헤더 (마커 일관성 깨기)
- 카드 제목 그대로 복붙
- "왜" 없는 변경 나열
- PR에서 보이는 정보(파일 목록·stat) 댓글에 중복

### 완료 댓글 포맷 (개정)
```
✅ DCTC-<번호> 완료 — <결과 한 줄, 고유명사 포함>
🔗 PR: <url>

📝 변경 요약
• <무엇> — <왜/효과>   (3~5개)

💡 결정: <대안 대비 왜, 한 줄>   ← 판단 갈렸을 때만, 평소 생략
후속: <있을 때만>
```
- 마커: `✅ / 🔗 / 📝 / 💡(선택) / 후속(선택)`
- **`📂 영향 범위` 마커 제거** — PR의 `git diff --stat`에서 확인 가능하므로 댓글에서 중복 제거.
- 본문 **10줄 이내** 권장 (이어붙이기 블록은 별도).
- 표·코드 블록·전체 파일 목록 금지 (PR/description에서 확인).

### description 플랜 포맷 (AI스러움 제거)
- 목표: 사용자 언어로 1~2줄. AI 재진술·장황한 배경 금지.
- 단계: "할 일" 동사구 + 검증 기준. 내부 사고과정·도구 나열 금지.
- 영향 파일: 경로만. 추측성 "예상" 남발 금지.
- 분량: 목표+단계+영향파일이 **화면 한 눈**. 넘으면 PR/별도 문서로.

### before/after 예시
규칙 파일에 "나쁜 댓글 → 좋은 댓글" 1쌍을 포함해 기준을 눈으로 확인 가능하게 한다.

## 검증 기준 (통과 정의)
1. `core/CLAUDE.md` 인덱스에 규칙이 등록되어 on-demand 로딩 경로 성립.
2. 두 커맨드/스킬에 `📂 영향 범위` 잔존 없음, 포맷은 규칙 단일 소스 참조.
3. 규칙 파일에 before/after 예시 1쌍 존재.
4. 기존 운영 디테일(이어붙이기, main 직접 PR 금지, 중복 PR 방지) 손실 없음.

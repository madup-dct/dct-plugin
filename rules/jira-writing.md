# Jira Writing (지라 라이팅)

Jira 댓글·description 을 "협업자가 다시 읽을 가치가 있게" 쓰는 규칙.
AI 라이팅의 함정(과정 나열·세션 로그 복붙)을 막고, 독자가 **왜·결론·근거**를 즉시
파악하고 **나중에 검색**할 수 있게 한다. `/dct-complete`, `dct-jira-workflow` 스킬이 참조한다.

원칙 5는 **모든 Jira 글에 공통**이고, 형식(표·헤더 허용 여부)은 문서 종류에 따라 갈린다(아래 "문서 종류").

## 원칙 5

1. **결론 먼저** — 첫 줄은 결과("무엇이 어떻게 됐다"). 과정·배경 narration 금지.
2. **왜 > 무엇** — 변경 bullet은 `<무엇> — <왜/효과>` 한 문장. 코드 보면 아는 "무엇"만 적지 않는다.
3. **검색되게** — 첫 줄·bullet에 고유명사(모듈/기능/에러명)를 넣는다. 카드 제목 복붙 금지.
4. **스캔 가능** — 마커·순서 고정, 산문체 금지. 5초 안에 구조 파악되게.
5. **세션 로그 ≠ 문서** — AI가 한 모든 일을 옮기지 않는다. PR에서 확인 가능한 것(변경 파일 목록, diff stat)은 댓글에 중복하지 않는다. 독자가 행동·판단에 쓸 것만 남긴다.

## 안티패턴 (하지 말 것)

| 하지 말 것 | 대신 |
|------------|------|
| 세션 히스토리·도구 호출·내부 사고 나열 | 결과와 판단 근거만 |
| 매번 다른 헤더 사용 | 마커·순서 고정 |
| 카드 제목 그대로 복붙 | 실제 변경 기반 재작성 |
| "왜" 없이 변경만 나열 | `<무엇> — <왜>` 한 문장 |
| PR에서 보이는 정보(파일 목록·stat) 댓글에 복붙 | PR 링크로 위임 |

## 문서 종류 — 형식 분기

원칙 5는 공통. **형식(표·헤더 허용 여부)만** 종류에 따라 다르다.

| 종류 | 어디서 읽힘 | 표·헤더 | 형식 |
|------|------------|---------|------|
| **A. 완료 댓글·진행 업데이트** | Slack 연동 채널 미러 | **금지** (Slack에서 깨짐) | 마커 5종 + bullet, 10줄 이내 |
| **B. 조사·분석·설계 보고** | Jira 에서 직접 읽음 | **권장** (스캔성↑) | 헤더 + 표 + 우선순위 라벨 |

판단 기준: **Slack 채널로 미러되면 A**(`/dct-complete` 산출물이 여기 해당), **Jira 에서 직접 펼쳐 읽는 깊은 보고면 B**.

## 형식 A — 완료 댓글·진행 업데이트 (Slack 미러)

```
✅ DCTC-<번호> 완료 — <결과 한 줄, 고유명사 포함>
🔗 PR: <url>   (또는: 생성 안 함 / 이미 병합됨)

📝 변경 요약
• <무엇> — <왜/효과>   (3~5개)

💡 결정: <대안 대비 왜 이 방향, 한 줄>   ← 판단이 갈렸을 때만, 평소 생략
후속: <있을 때만 한 줄, 없으면 줄 생략>
```

- 마커: `✅ / 🔗 / 📝 / 💡(선택) / 후속(선택)`, 재작업 구분은 `🔁`. 다른 헤더(`##`)·코드 블록·표 금지.
- 본문 **10줄 이내** 권장. 이어붙이기(재작업) 블록은 별도 카운트.
- 영향 범위·변경 파일 전체 목록·검증 결과는 **PR 본문**에서 확인 (댓글에 중복 금지).
- 재작업 시: 기존 완료 댓글을 수정해 `---` + `🔁 추가 작업 (YYYY-MM-DD, KST)` 블록으로 이어붙인다. 새 댓글 생성 금지.

## description 플랜 포맷

- 목표: 사용자 언어로 1~2줄. AI 재진술·장황한 배경 금지.
- 단계: "할 일" 동사구 + 검증 기준. 내부 사고과정·도구 나열 금지.
- 영향 파일: 경로만. 추측성 "예상" 남발 금지.
- 분량: 목표+단계+영향파일이 **화면 한 눈**. 넘으면 PR/별도 문서로.

## 형식 A 예시 (before → after)

**나쁨** (세션 로그·과정 나열·왜 없음):
```
✅ 작업 완료 요약
로그인 기능을 수정했습니다. 먼저 코드를 읽고, AuthService를 분석한 뒤
여러 파일을 수정했습니다. LoginForm.tsx, authService.ts, api.ts 를 변경했고
테스트를 돌려서 통과했습니다. 자세한 건 PR을 봐주세요.
```

**좋음** (결론 먼저·왜·검색 키워드):

## 형식 B — 조사·분석·설계 보고 (Jira 직독)

Jira 카드에서 직접 펼쳐 읽는 깊은 보고(전수조사, 아키텍처 분석, 설계 검토). Slack 미러 대상이 아니므로 표·헤더를 적극 쓴다. 원칙 5(특히 결론 먼저·검색되게·스캔 가능)는 그대로 적용.

- **결론 먼저**: 첫 헤더·첫 줄에 "그래서 결론이 뭔가". (예: "하드코딩이 맞고, 기존 Merchant 컬럼으로 대체 가능")
- **우선순위 라벨**: 항목이 여럿이면 `P0/P1/P2` 등 급한 순. 독자가 "뭘 먼저"를 5초에 파악.
- **표로 압축**: 반복 구조(위치·증상·조치 등)는 산문 대신 표. 열 3~5개.
- **노이즈 분리**: 오탐·제외 항목은 별도 섹션으로 빼서 진짜 이슈와 안 섞는다.
- **검색 키워드**: 모듈·컬럼·상수·레포명 등 고유명사를 본문에 박는다(원칙 3).
- 스키마·타 레포 영향은 `⚠` 로 눈에 띄게.

참고 모범: DCTC-2553 전수조사 댓글 — P0~P2 우선순위 표 + 오탐(43/59) 분리 + 고유명사(`CatalogProductAudienceDialog`, `partner_business_id`, `fb_app_id`) 다수.
